## Bugs Fixed

### Bug #1: SQR Function - Wrong Address for Exponent ✅ FIXED

**Severity:** High
**Impact:** All SQR() function calls returned wrong values

**Symptom:**
- SQR(4) returned 1.0 instead of 2.0
- SQR(9) returned 0.75 instead of 3.0
- SQR(2) returned 0.707... (1/√2) instead of 1.414...

**Root Cause:**
Line 3820 in scelbal.mac was using SCRATCH_1F (address 0x004E) to access the SQR approximation exponent, but SCRATCH_1F was at FPACC_SAVE+4. The exponent is actually at FPACC_SAVE+3 (address 0x004D).

**Fix:**
```assembly
; OLD (WRONG):
LXI H,SCRATCH_1F    ; Change L to point to SQR APPROX exponent.

; NEW (CORRECT):
LXI H,FPACC_SAVE+3  ; Change L to point to SQR APPROX exponent
```

**File:** scelbal.mac, line 3820

**Verification:**
- SQR(4) => 2.0 ✓
- SQR(9) => 3.0 ✓
- SQR(2) => 1.414213 ✓
- SQR(16) => 4.0 ✓
- SQR(25) => 5.0 ✓
- SQR(100) => 10.0 ✓

### Bug #3: Test Infrastructure - Grep Pattern Issues ✅ FIXED

**Impact:** Test runner couldn't properly extract results

**Problems:**
1. Pattern `^[[:space:]]*[-0-9]` only matched ONE character
2. Didn't account for negative numbers (no leading space)
3. Didn't filter out debug output (matched wrong lines)
4. Didn't strip carriage returns

**Fix:**
```bash
# OLD:
result=$(... | grep -a "^[[:space:]]*[-0-9]" | head -1 | tr -d ' ')

# NEW:
result=$(... | grep -Ea "^[[:space:]]*-?[0-9][0-9.E+-]*"$'\r' | head -1 | tr -d ' \r')
```

**Improvements:**
- Matches full numbers including decimals and scientific notation
- Handles negative numbers correctly
- Filters lines ending with `\r` to avoid debug output
- Strips both spaces and carriage returns

**Files Updated:**

- quick_test.sh
- All test scripts

## Technical Notes

### SQR Function Memory Layout
```
FPACC_SAVE:     0x004A  (4 bytes: LSW, NSW, MSW, EXP)
FPACC_SAVE+0:   0x004A  LSW (Least Significant Word)
FPACC_SAVE+1:   0x004B  NSW (Next Significant Word)
FPACC_SAVE+2:   0x004C  MSW (Most Significant Word)
FPACC_SAVE+3:   0x004D  EXP (Exponent) ← CORRECT ADDRESS
SCRATCH_1F:     0x004E  (separate variable) ← WRONG!
```

The bug was using SCRATCH_1F (0x004E) instead of FPACC_SAVE+3 (0x004D) to access the exponent, causing the exponent to be read from and written to the wrong memory location.
